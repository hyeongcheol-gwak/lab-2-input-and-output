# `dirtree.c` 코드 분석 및 개선 리뷰 (README 규격 검토)

`README.md`의 세부 규격과 `dirtree.c`의 구현 로직을 논리적으로 대조하며 검토한 결과입니다 (`check.sh` 등의 외부 툴에 의존하지 않고 소스 코드 자체를 바탕으로 도출되었습니다).

전반적으로 프로그램의 기능적 핵심(재귀 순회, 정규식 기반 AST 패턴 매칭, 심볼릭 링크 및 노드 처리 등)은 지침에 맞게 아주 잘 구현되어 있습니다. 다만 **출력 형식(Spacing)** 및 **예외 상황 출력 로직**에서 규격을 완벽히 충족하기 위한 2가지 핵심 수정 사항이 발견되었습니다.

---

## 1. 입출력 형식 (I/O Formatting) 검토 결과

### ✅ 완전히 올바르게 구현된 부분 (Pass)
- **컬럼 제약 사항 준수:** 
  `print_formats[4]`에 명시된 `%-54s  %8.8s:%-8.8s  %10llu  %8llu    %c\n` 포맷은 길이를 환산했을 때 54 + 2 + 8 + 1 + 8 + 2 + 10 + 2 + 8 + 4 + 1 = **총 100글자**로, 요구사항의 길이와 완벽하게 일치합니다.
- **사용자/그룹 트렁케이션 및 정렬:**
  요구사항에 명시된 `Right aligned`(사용자), `Left aligned`(그룹) 조건을 포맷 지정자 `%8.8s`, `%-8.8s`를 통해 깔끔하게 처리했습니다.
- **파일명 말줄임 처리 (Truncation):** 
  들여쓰기(Indent) 폭이 포함된 상태에서 54를 초과할 경우 `max_name_len - 3`만큼 잘라내고 `...`을 붙이는 수식은 파일명이 한계를 넘었을 때 완벽하게 동작합니다.

### ❌ 개선이 필요한 부분 (Summary Line 여백 버그)
- **원인 분석:** 
  `print_summary` 함수의 서식 문자열이 다음과 같이 작성되어 있습니다.
  `printf("%-68s %14llu %9llu\n", buf, stats->size, stats->blocks);`
- **논리적 오류:**
  해당 포맷은 요약문을 68칸 공간에 할당하고, 그 다음 공간들 사이에 공백(` `)을 각각 1칸씩만 추가합니다.
  하지만 README에 명시된 출력 컬럼 구조상 파일의 크기(Size)는 85번째 칸에서 끝나야 하고 블록(Blocks)은 95번째 칸에서 끝나야 합니다. 
  즉, 요약 문장 길이는 68이므로 `68 + 여백 + 14 = 85` 가 성립해야 합니다. 따라서 68칸의 문자열 뒤에는 1칸이 아니라 **3칸의 공백**이 들어가야 Size 컬럼 위치가 정확히 맞춰집니다. (현재 코드대로면 출력물이 왼쪽으로 2칸씩 쏠려 출력됩니다.)
- **해결 방안:** 
  `printf("%-68s   %14llu %9llu\n", ...)` 로 서식의 공백을 3칸으로 수정해야 합니다.

## 2. 에러 출력 (Error Reporting) 로직 검토 결과

README에 따르면 오류 시 다음과 같이 출력되어야 합니다.

- **존재하지 않는 폴더 접근 시 (`lstat` 실패):** 파일/폴더 정보 출력 없이 다음 줄에 에러 표기
- **권한 거부 시 (`opendir` 실패):** **폴더 본인의 정보(유저/사이즈 등)는 정상 출력한 뒤**, 내부 항목들 대신 계층을 들여쓰기하여 에러 표기

### ❌ 코드의 구조적 문제점 (Bug)
- 현재 `TreeNode` 구조체는 어떠한 에러가 나든 `node->has_error = 1;`로 동일한 필드에 기록합니다.
- 이에 따라 `print_tree` 함수는 `else if (node->has_error && node->error_msg)` 분기문을 태우는데, 이 경우 해당 노드의 정보(Stats)를 출력하는 정상적인 `else` 블록(크기/권한 출력)을 완전히 우회하게 됩니다.
- 그 결과, 파일 정보는 읽을 수 있지만(`lstat` 성공), 폴더 내부 진입만 제한된(`opendir` 실패, 예: Permission Denied) 대상의 경우 **"자신의 파일 정보까지 통째로 누락"**되는 현상이 생깁니다.

### 💡 구조 개선 방안
에러 처리를 명확히 나누고 로직을 분리하는 구조적 설계 개선이 필요합니다. 
`TreeNode`에 에러 종류를 구분하는 구조가 필요합니다.
```c
// 구조체 변경 제안 (두 개의 분리된 플래그)
int lstat_error;    // 정보 조회 자체가 실패 (본인 정보 없음)
int opendir_error;  // 폴더 내부 진입 실패 (본인 정보는 존재)
```
이에 맞춰 `print_tree` 출력부를 분할할 수 있습니다.
1. `lstat_error`일 때만 이름/에러를 함께 출력하고 통계 정보(Stats)는 생략.
2. `opendir_error`일 경우 정상적으로 폴더의 정보(User/Size/Block 등)를 우선 출력한 뒤, `indent + 2` 칸의 들여쓰기와 함께 `ERROR: Permission denied`를 다음 줄에 출력.

## 3. 구조적 아키텍처 및 추가 성능 개선 제안

**① 자체 정규식 엔진의 메모리 구조 확장성 개선**
- 로직 자체는 훌륭하게 Abstract Syntax Tree(AST)와 Memoization 백트래킹을 활용하고 있으나, 문법 위반 시 `panic` 함수를 호출해 **곧바로 프로그램을 종료(`exit`)**하게 설계되어 있습니다. 
- 단발성 Lab 과제로는 충분하지만, 확장성 및 라이브러리화를 고려한다면 에러 발생 시 동적 할당한 트리를 `free`로 반환한 뒤 오류 코드(`EXIT_FAILURE`)를 상위 함수로 리턴하는 "Clean Exit" 구조가 더 좋은 설계입니다.
- 함수 내에서 글로벌 변수(`node_id_counter`, `p_curr`)로 파싱 상태를 제어하는 부분 역시 다수의 패턴을 처리하거나 멀티스레드 환경을 고려한다면 `ParsingContext` 구조체 등으로 감싸서 매개변수로 주고받는 캡슐화(Encapsulation)를 권장합니다.

**② 심볼릭 링크 구조체 검사 로직 최적화**
- `dirent_compare` 정렬 함수를 보면 오직 직관적인 `is_dir` 값 하나만 판단하여 폴더/파일 간의 우선순위를 정렬하고 있습니다. 
- 현재의 목적에는 부합하나 `st_mode` 기반의 공통 비교 함수를 만들면 이후 블록 디바이스 타입 등 다른 타입의 파일 제어 로직이 요구사항에 추가될 때 훨씬 정교하게 유지보수 할 수 있습니다.
